## § 4.7. Создаём регистр «ОстаткиТоваров» и справочник «Склады»

### Открытие

К концу Модуля 3 документ «ПоступлениеТоваров» хранил склад как строку в реквизите шапки: бухгалтер вводил «Основной» вручную. Это неудобно и ненадёжно — легко ошибиться в написании, нельзя вести справочник складов.

В этом параграфе мы:
1. Создадим справочник «Склады»
2. Изменим реквизит «Склад» в документе: вместо строки — ссылка на справочник
3. Создадим регистр накопления «ОстаткиТоваров»
4. Подключим «ПоступлениеТоваров» как регистратора
5. Проверим результат

### Шаг 1: создаём справочник «Склады»

В дереве метаданных — ветка «Справочники» → правой кнопкой → «Добавить».

**Вкладка «Основные»:**

```
Имя:     Склады
Синоним: Склады
```

**Вкладка «Данные»:**

Реквизиты не добавляем — для начала хватит стандартного «Наименование».

Нажмите ОК.

```
Справочник «Склады»
└── Наименование (Строка)   ← стандартный реквизит
```

### Шаг 2: меняем тип реквизита «Склад» в документе

Откройте документ «ПоступлениеТоваров» → вкладка «Данные» → раздел «Реквизиты».

Найдите реквизит «Склад» с типом «Строка». Измените тип на «СправочникСсылка.Склады».

```
Было:    Склад — тип: Строка
Стало:   Склад — тип: СправочникСсылка.Склады
```

Теперь поле «Склад» в документе — это не свободный текст, а выбор из справочника. При вводе документа бухгалтер выберет склад из списка, а не напечатает название.

Нажмите ОК.

### Шаг 3: создаём регистр накопления «ОстаткиТоваров»

В дереве метаданных — ветка «Регистры накопления» → правой кнопкой → «Добавить».

**Вкладка «Основные»:**

```
Имя:     ОстаткиТоваров
Синоним: Остатки товаров
Вид:     Остатки   ← выбрать из списка (не «Обороты»)
```

«Вид: Остатки» означает, что регистр поддерживает разделение движений на Приход и Расход и умеет вычислять остатки. Вид «Обороты» — только для подсчёта оборотов без разделения на направление (используется реже).

**Вкладка «Данные»:**

В разделе «Измерения» добавьте:

```
Товар  — тип: СправочникСсылка.Товары
Склад  — тип: СправочникСсылка.Склады
```

В разделе «Ресурсы» добавьте:

```
Количество — тип: Число, точность 0
```

Нажмите ОК.

```
Регистр накопления «ОстаткиТоваров»
├── Вид: Остатки
├── Измерения:
│   ├── Товар   (СправочникСсылка.Товары)
│   └── Склад   (СправочникСсылка.Склады)
└── Ресурсы:
    └── Количество (Число)
```

### Шаг 4: подключаем «ПоступлениеТоваров» как регистратора

Откройте документ «ПоступлениеТоваров» → вкладка «Движения».

Нажмите «Добавить», выберите «Регистр накопления "ОстаткиТоваров"».

Укажите **Вид движения: Приход** — поступление товаров всегда увеличивает остаток.

Нажмите «Конструктор движений». В открывшемся диалоге укажите соответствие полей:

```
Измерение «Товар»        ← поле «Товар» из табличной части «Товары»
Измерение «Склад»        ← реквизит шапки «Склад»
Ресурс «Количество»      ← поле «Количество» из табличной части «Товары»
```

Нажмите ОК. Платформа создаст необходимые настройки.

```
Документ «ПоступлениеТоваров»
└── Движения:
    └── Регистр накопления «ОстаткиТоваров» → Приход
        ├── Товар     ← из строки табличной части
        ├── Склад     ← из шапки документа
        └── Количество ← из строки табличной части
```

### Шаг 5: сохранить и проверить

Нажмите F7 (сохранить конфигурацию), затем F8 (обновить базу данных).

**В пользовательском режиме:**

Откройте справочник «Склады» и добавьте хотя бы один склад: «Основной».

Создайте новый документ «ПоступлениеТоваров»:
- Контрагент: выберите из справочника
- Склад: выберите «Основной» из справочника «Склады»
- Табличная часть: добавьте несколько товаров с количеством

Нажмите «Провести».

Откройте регистр накопления «ОстаткиТоваров» (через «Все функции»). Убедитесь, что появились движения с видом «Приход».

### Итоговая схема

```
Конфигурация «Канцтовары»
│
├── Справочники
│   ├── Товары
│   │   └── Реквизиты: Наименование, Артикул, Единица
│   ├── Контрагенты
│   │   └── Реквизиты: Наименование, ИНН, Телефон
│   └── Склады                           ← новый
│       └── Реквизиты: Наименование
│
├── Документы
│   ├── ПоступлениеТоваров
│   │   ├── Реквизиты шапки: Контрагент (→Контрагенты), Склад (→Склады)
│   │   ├── Табличная часть «Товары»: Товар (→Товары), Количество, Цена, Сумма
│   │   └── Движения → ОстаткиТоваров (Приход)   ← новое
│   └── ПриказОбИзмененииЦен
│       ├── Табличная часть «Товары»: Товар (→Товары), Цена
│       └── Движения → ЦеныТоваров
│
├── Регистры сведений
│   └── ЦеныТоваров
│       ├── Режим: Подчинённый регистратору
│       ├── Периодичность: По позиции регистратора
│       ├── Измерения: Товар (→Товары)
│       └── Ресурсы: Цена (Число)
│
└── Регистры накопления                   ← новая ветка
    └── ОстаткиТоваров                    ← новый
        ├── Вид: Остатки
        ├── Измерения: Товар (→Товары), Склад (→Склады)
        └── Ресурсы: Количество (Число)
```

### Главное

Учёт остатков товаров потребовал трёх изменений: нового справочника «Склады» (чтобы склад был объектом, а не строкой), нового регистра накопления «ОстаткиТоваров» (с измерениями Товар и Склад, ресурсом Количество), и настройки движений в «ПоступлениеТоваров» с видом «Приход». Теперь каждое проведённое поступление автоматически обновляет остатки — без ручного ввода и без риска ошибки.

### Контрольные вопросы

1. Почему реквизит «Склад» в документе лучше хранить как ссылку на справочник, а не как строку?
2. Что означает «Вид: Остатки» у регистра накопления — чем он отличается от вида «Обороты»?
3. Что будет с данными в базе, если изменить тип реквизита «Склад» с «Строка» на «СправочникСсылка.Склады» в уже работающей базе с данными?
4. Зачем в Конструкторе движений нужно указывать соответствие полей — платформа не может сделать это автоматически?
5. Что нужно добавить в конфигурацию, чтобы «ПоступлениеТоваров» учитывало не только количество, но и сумму поступления?

### Упражнения

**Сделай в Конфигураторе.** Выполните все четыре шага из этого параграфа. Добавьте в справочник «Склады» два склада: «Основной» и «Розничный». Проведите два документа поступления — на разные склады. Откройте регистр «ОстаткиТоваров» и убедитесь, что движения разделены по складам.

**Посмотри и ответь.** После того как документ проведён — откройте его снова и измените количество в одной строке. Перепроведите. Откройте регистр — что изменилось в движениях? Совпадает ли новый остаток с ожидаемым? Что будет, если отменить проведение документа полностью?

**Спроектируй.** В «Канцтовары» скоро появится документ «РасходнаяНакладная» (продажа клиенту). Опишите, что нужно настроить в этом документе, чтобы он корректно уменьшал остатки в регистре «ОстаткиТоваров». Что изменится в структуре конфигурации?

### Ответы и указания

*Вопрос 3.* При изменении типа реквизита платформа потребует обновить структуру базы данных (F8). Ранее введённые значения строкового реквизита будут потеряны или преобразованы — в зависимости от версии платформы и состояния базы. В учебной базе без реальных данных это не проблема; в рабочей базе — требует предварительного переноса данных.

*Вопрос 4.* Платформа знает *куда* писать (какой регистр), но не знает, какое поле документа соответствует какому полю регистра: у документа может быть несколько табличных частей и реквизитов с похожими именами. Конструктор движений — это инструмент явного указания этого соответствия.

*Упражнение «Спроектируй».* В документе «РасходнаяНакладная» на вкладке «Движения» нужно добавить регистр накопления «ОстаткиТоваров» с видом движения «Расход». В Конструкторе движений задать соответствие: Товар из строки документа → измерение Товар, Склад из шапки → измерение Склад, Количество → ресурс Количество. Новых объектов метаданных добавлять не нужно — регистр уже существует.
