## § 7.7. Другие объекты дерева: константы, планы обмена, журналы документов, функциональные опции

### Открытие

В дереве реальной конфигурации вы встретите объекты, которые мы не разбирали подробно. Они не требуют глубокого погружения, но нужно понимать их назначение — чтобы не теряться, когда они попадутся на глаза.

В этом параграфе — четыре таких объекта. Для каждого: что это, зачем нужно, как выглядит в дереве.

### Константы

Константа — это одно значение, которое хранится на уровне всей конфигурации и не меняется в ходе обычных операций.

Примеры: наименование организации, ставка НДС по умолчанию, текущий финансовый год, адрес электронной почты для уведомлений.

```
Константы
├── НаименованиеОрганизации   = «ООО Канцтовары»
├── СтавкаНДС                 = 20
└── АдресПоддержки            = «support@kantctovary.ru»
```

Константа отличается от реквизита справочника тем, что она одна на всю систему — не привязана ни к какому объекту. Изменить её может только администратор через специальный интерфейс, а не рядовой пользователь в ходе работы.

> Если нужно хранить одно общесистемное значение — константа. Если значение у каждого объекта своё — реквизит.

### Планы обмена

Конфигурация редко живёт в изоляции. Она обменивается данными с другими системами: с интернет-магазином, с онлайн-кассой, с базой головного офиса, с мобильным приложением.

План обмена — это объект, который описывает: с кем обменивается конфигурация и какими данными. Каждый узел плана обмена — это один «собеседник» (другая база, внешняя система).

```
План обмена «СинхронизацияСГоловнымОфисом»
├── Узлы: ГоловнойОфис, Филиал_Москва, Филиал_СПб
└── Состав: Справочники (Товары, Контрагенты), Документы (Реализация)
```

При чтении чужого дерева: если есть «Планы обмена» — эта конфигурация не изолирована, она часть распределённой системы или интегрирована с чем-то ещё.

### Журналы документов

В большой конфигурации документов десятки: поступления, реализации, перемещения, возвраты, акты, накладные. Открывать каждый тип по отдельности неудобно.

Журнал документов — это объект, который объединяет несколько типов документов в один список для просмотра. Это не отдельное хранилище — данные остаются в своих объектах. Журнал — только представление.

```
Журнал «ВсеДокументыПоТоварам»
├── ПоступлениеТоваров
├── РеализацияТоваров
├── ПеремещениеТоваров
└── СписаниеТоваров

Пользователь открывает журнал — видит все документы
по товарам в хронологическом порядке, независимо от типа.
```

При чтении чужого дерева: журналы говорят о том, как организован интерфейс, а не о хранении данных. Если журнал объединяет несколько типов документов — значит, в бизнес-логике они рассматриваются как одна группа операций.

### Функциональные опции

В крупных конфигурациях не весь функционал нужен всем клиентам. Один клиент ведёт бухгалтерию, другой — нет. Один использует производственный учёт, другой — только торговлю.

Функциональная опция — это переключатель, который включает или выключает части функционала конфигурации. Когда опция выключена — соответствующие объекты, поля и команды скрываются из интерфейса.

```
Функциональные опции
├── ИспользоватьБухгалтерскийУчёт   = Нет
│   (скрывает регистр бухгалтерии, план счетов, соответствующие отчёты)
├── ИспользоватьПроизводство         = Нет
│   (скрывает производственные документы и регистры)
└── ИспользоватьРозничнуюТорговлю   = Да
    (показывает кассовые документы и отчёты)
```

Это объясняет важное наблюдение: открываете две базы с одной конфигурацией — а они выглядят по-разному. Не потому что конфигурации разные, а потому что функциональные опции настроены по-разному.

При чтении дерева: если в конфигурации есть «Функциональные опции» — это признак универсальной системы, рассчитанной на разные варианты использования.

### Главное

Четыре объекта, которые встречаются в любой серьёзной конфигурации:

- **Константы** — одно общесистемное значение, не привязанное к объектам
- **Планы обмена** — описание обмена данными с другими системами
- **Журналы документов** — объединённый вид нескольких типов документов для удобства
- **Функциональные опции** — переключатели, которые включают и выключают части функционала

### Контрольные вопросы

1. Чем константа отличается от реквизита справочника?
2. Что означает наличие «Планов обмена» при чтении незнакомой конфигурации?
3. Журнал документов — это отдельное хранилище данных или только представление?
4. Почему две базы с одной конфигурацией могут выглядеть по-разному?
5. Вы открываете конфигурацию и видите реквизит в документе, который недоступен для заполнения. Каким объектом это может быть обусловлено?

### Упражнения

**Найди в Конфигураторе.** Откройте демо-базу. Найдите ветку «Константы» — сколько констант? Найдите ветку «Журналы документов» — какие типы документов объединяет каждый журнал? Найдите «Функциональные опции» — есть ли они, и что они включают или выключают?

**Посмотри и ответь.** Найдите в демо-базе журнал документов и посмотрите, какие типы документов в него включены. Что общего у этих документов с точки зрения бизнес-логики? Если добавить новый тип документа в конфигурацию — попадёт ли он в журнал автоматически или нужна отдельная настройка?

**Спроектируй.** В «Канцтовары» хотят добавить интеграцию с интернет-магазином: заказы из интернет-магазина должны попадать в базу как документы, а остатки — передаваться обратно на сайт. Какой объект метаданных нужно создать? Что в нём нужно описать?

### Ответы и указания

*Вопрос 3.* Журнал документов — только представление. Данные хранятся в своих объектах (документах), журнал просто показывает их вместе в одном списке. Удаление журнала не удаляет документы.

*Вопрос 5.* Реквизит может быть недоступен из-за функциональной опции: она выключена, и реквизит скрыт или заблокирован. Также возможно ограничение по роли (нет права на запись).

*Упражнение «Спроектируй».* Нужен план обмена. В нём описать: узлы (конфигурация 1С и интернет-магазин), состав данных для отправки (остатки из регистра накопления) и получения (заказы → документы). Детальная настройка обмена — отдельная тема за рамками книги.

### Мостик

Теперь вы знаете весь основной словарный запас дерева метаданных. Итог модуля — краткая карта: какой объект за что отвечает.
