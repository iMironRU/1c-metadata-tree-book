# Модуль 7. Другие объекты дерева

## § 7.1. Перечисления: список, закрытый навсегда

### Открытие

В предыдущих модулях мы изучали объекты, которые хранят данные: справочники хранят классификаторы, документы фиксируют события, регистры накапливают состояния. В Модуле 7 мы познакомимся с остальными объектами дерева — теми, которые реже оказываются в центре внимания, но встречаются в любой реальной конфигурации.

> Цель модуля — научиться узнавать объекты в дереве и понимать, для чего они созданы. Мы не осваиваем бухгалтерию, расчёт зарплаты или безопасность как профессиональные дисциплины — это темы отдельных курсов. Здесь достаточно понять назначение, а не внутреннюю механику.

Начнём с самого простого: перечисления.

Откройте Конфигуратор. В дереве метаданных найдите ветку **«Перечисления»**. В демо-базе там будет несколько объектов — конкретный набор зависит от конфигурации, важен не он, а навык чтения. Разверните любой — и увидите не реквизиты, а список значений. Это и есть перечисление.

### Что такое перечисление

Перечисление — это закрытый список значений, заданный один раз при разработке. Закрытый — значит пользователь не может добавить новое значение, изменить существующее или удалить. Список фиксирован.

Пример: времена года. Зима, Весна, Лето, Осень — и больше ничего. Этот список ограничен самой реальностью: разработчик знает его заранее и знает, что он не изменится.

```
Перечисление «ВременаГода»
├── Зима
├── Весна
├── Лето
└── Осень
```

Другой пример: виды оплаты в торговой системе — наличные, банковская карта, безналичный перевод. Этот список тоже ограничен, но по другой причине: не реальностью, а решением разработчика. Он реализовал именно эти три способа — и зафиксировал их в перечислении. Если понадобится четвёртый, придётся менять конфигурацию.

> Источник ограниченности перечисления бывает двух видов: реальность сама ограничивает список (времена года, стороны света), или разработчик принял решение поддерживать именно эти варианты (виды оплаты, статусы заказа).

### Чем перечисление отличается от справочника

На первый взгляд похоже: и там, и там — список значений. Разница принципиальная.

```
Справочник                          Перечисление
──────────────────────────────────  ──────────────────────────────────
Пользователь добавляет элементы     Только разработчик меняет список
Список открыт                       Список закрыт
Элементы — это данные               Значения — часть структуры
Можно добавить реквизиты            Реквизитов нет совсем
```

Справочник «Склады» — пользователь открывает его и добавляет новый склад. Перечисление «ВременаГода» — пользователь не может добавить пятое время года.

Важно: **у перечислений нет реквизитов**. У справочника «Товары» есть Артикул, Единица, Наименование. У перечисления — только сами значения, и всё. Нельзя добавить к значению «Зима» поле «Описание» или «Порядковый номер». Перечисление — это именно список, не больше.

### Предопределённые значения справочника — это не то же самое

В справочниках есть похожий инструмент: предопределённые элементы. Разработчик создаёт элемент прямо в конфигурации — например, «Основной склад» в справочнике «Склады». Этот элемент нельзя удалить в пользовательском режиме.

Но это не перечисление: пользователь по-прежнему может добавить рядом другие склады. Предопределённый элемент — это зафиксированная «точка отсчёта» внутри открытого списка. Перечисление — это полностью закрытый список без исключений.

### Где используются перечисления

Перечисление появляется там, где реквизит объекта должен принимать строго одно из нескольких фиксированных значений. Например:

```
Справочник «Контрагенты»
└── Реквизит «ВидКонтрагента» (тип: ПеречислениеСсылка.ВидыКонтрагентов)
    Возможные значения: Поставщик / Покупатель / Сотрудник / Прочее
```

Тип реквизита в этом случае — не строка и не число, а ссылка на значение перечисления. Это означает: реквизит может принять только одно из заранее перечисленных значений.

### Главное

Перечисление — закрытый список значений без реквизитов. Пользователь не может его изменить. Список ограничен либо самой реальностью (времена года, стороны света), либо решением разработчика (поддерживаемые способы оплаты). Перечисление отличается от справочника тем, что его список зафиксирован в конфигурации и не расширяется в пользовательском режиме. Реквизитов у перечислений нет — только сами значения.

### Контрольные вопросы

1. Чем перечисление отличается от справочника с точки зрения пользователя?
2. Почему у перечисления нет реквизитов?
3. Какие два источника ограниченности списка перечисления существуют? Приведите пример для каждого.
4. Что такое предопределённый элемент справочника и чем он отличается от значения перечисления?
5. Реквизит документа «ВидОперации» имеет тип «ПеречислениеСсылка.ВидыОпераций». Что это означает для пользователя при заполнении документа?

### Упражнения

**Найди в Конфигураторе.** Откройте демо-базу. Разверните ветку «Перечисления». Найдите любое перечисление и посмотрите его значения. Затем найдите документ или справочник, в реквизитах которого используется это перечисление (тип реквизита будет содержать слово «Перечисление»). Опишите: что означает этот реквизит с точки зрения бизнес-логики?

**Посмотри и ответь.** Найдите в демо-базе справочник, у которого есть предопределённые элементы (в дереве они отображаются особым значком). Чем поведение предопределённого элемента отличается от обычного в пользовательском режиме? Что сломается в бизнес-логике, если попытаться удалить предопределённый элемент — и почему платформа это не позволяет?

**Спроектируй.** В «Канцтовары» решили добавить реквизит «СтатусЗаказа» для нового документа «ЗаказПокупателя». Возможные статусы: Новый, Подтверждён, Отгружен, Отменён. Обоснуйте: почему статусы стоит реализовать перечислением, а не справочником? При каком условии их всё же стоило бы сделать справочником?

### Ответы и указания

*Вопрос 1.* Пользователь не может добавить новое значение в перечисление — список зафиксирован при разработке. В справочник пользователь добавляет новые элементы в любой момент.

*Вопрос 4.* Предопределённый элемент — это элемент справочника, созданный разработчиком и защищённый от удаления. Но рядом с ним пользователь может добавить другие элементы — список остаётся открытым. Значение перечисления — часть закрытого списка; ни добавить, ни удалить его в пользовательском режиме невозможно.

*Упражнение «Спроектируй».* Статусы стоит реализовать перечислением: их набор фиксирован логикой бизнес-процесса, добавление нового статуса требует изменения логики системы (а значит, и конфигурации). Справочник был бы уместен, если бы статусы управлялись пользователем — например, в системе, где каждая компания сама настраивает свои этапы работы.

### Мостик

Перечисления — самый простой объект дерева. Теперь перейдём к самому сложному из тех, что встречаются в учётных системах: регистрам бухгалтерии и плану счетов.
