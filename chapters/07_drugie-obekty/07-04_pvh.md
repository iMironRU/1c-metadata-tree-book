## § 7.4. Планы видов характеристик: прилагательные пользователя

### Открытие

В Модуле 1 мы говорили о реквизитах справочника как о «прилагательных»: товар — красный, длиной 19 см, в упаковке по 12 штук. Разработчик знает заранее, какие прилагательные нужны, и создаёт под каждое отдельный реквизит.

Но что делать, если разработчик не знает заранее, какие характеристики понадобятся? Одна компания продаёт карандаши — ей важны цвет и длина. Другая продаёт ткани — ей важны состав и плотность. Третья — электронику: там модель, мощность, класс энергопотребления.

Жёстко прописать реквизиты для всех случаев невозможно. Для этого существует особый объект.

Откройте Конфигуратор. В дереве найдите **«Планы видов характеристик»**.

### Прилагательное, которое придумывает пользователь

Реквизит — это прилагательное, известное на этапе разработки: разработчик создал поле «Цвет», и оно есть у всех товаров навсегда.

План видов характеристик (ПВХ) — это механизм, который позволяет **пользователю** добавлять свои прилагательные уже после разработки. Разработчик оставляет «пустое место» для характеристик, а пользователь сам решает, что там будет.

```
Без ПВХ (реквизиты заданы разработчиком):
Товар «Карандаш HB»
├── Наименование: Карандаш HB
├── Артикул: K001
└── Единица: шт.
(Цвет или Длина? Только если разработчик их предусмотрел.)

С ПВХ (пользователь добавил характеристики сам):
Товар «Карандаш HB»
├── Наименование: Карандаш HB
├── Артикул: K001
├── Единица: шт.
├── [Цвет]: Чёрный         ← пользователь добавил эту характеристику
└── [Длина]: 19 см         ← и эту тоже
```

### Как устроен ПВХ

Объект «План видов характеристик» — это справочник возможных характеристик. Пользователь сам добавляет туда нужные ему виды: Цвет, Длина, Материал, Плотность — что угодно.

```
ПВХ «ВидыХарактеристикТоваров»
├── Цвет       (тип значения: Строка)
├── Длина      (тип значения: Число)
├── Материал   (тип значения: Строка)
└── Плотность  (тип значения: Число)
```

Значения характеристик хранятся отдельно — в специальном регистре сведений (тот самый инструмент из Модуля 3, только теперь его измерение указывает не на справочник, а на ПВХ). Запись в регистре: «Товар = Карандаш HB, Вид характеристики = Цвет, Значение = Чёрный».

### Почему это не то же самое, что добавить реквизит

Реквизит — часть структуры объекта. Добавить новый реквизит можно только изменив конфигурацию (это работа разработчика). ПВХ позволяет добавлять новые виды характеристик прямо в пользовательском режиме — без изменения конфигурации.

```
Реквизит                        Характеристика через ПВХ
──────────────────────────────  ──────────────────────────────────────
Структура зафиксирована         Пользователь расширяет сам
Нужно менять конфигурацию       Добавляется в пользовательском режиме
Поле есть у всех объектов       Характеристика есть только там, где нужна
Тип данных строго задан         Тип задаётся при создании вида характеристики
```

### Что видно в дереве при чтении конфигурации

Если в дереве есть «Планы видов характеристик» — значит, в конфигурации предусмотрено, что пользователь будет сам описывать свойства объектов. Обычно это сопровождается регистром сведений, который хранит конкретные значения характеристик.

Сигнал при чтении чужого дерева: «здесь гибкая модель данных — набор свойств объекта настраивается в процессе эксплуатации».

### Главное

План видов характеристик — механизм для «прилагательных пользователя»: характеристик объектов, набор которых неизвестен при разработке. Разработчик оставляет место, пользователь сам добавляет нужные виды (Цвет, Размер, Материал). Значения хранятся в отдельном регистре сведений. Отличие от реквизита: реквизит фиксируется разработчиком и присутствует у всех объектов данного типа; характеристика через ПВХ добавляется пользователем и присутствует только там, где нужна.

### Контрольные вопросы

1. В чём разница между реквизитом справочника и характеристикой через ПВХ?
2. Кто добавляет новые виды характеристик — разработчик или пользователь?
3. Где хранятся конкретные значения характеристик (например, «Цвет = Красный» для конкретного товара)?
4. Если в конфигурации есть «Планы видов характеристик» — о чём это говорит при чтении чужого дерева?
5. Почему нельзя просто добавить все нужные характеристики как реквизиты справочника «Товары»?

### Упражнения

**Найди в Конфигураторе.** Откройте демо-базу. Найдите план видов характеристик. Какие виды характеристик в нём перечислены? Найдите регистр сведений, в котором хранятся значения этих характеристик. Какие измерения и ресурсы у него есть?

**Посмотри и ответь.** Откройте один из товаров в пользовательском режиме демо-базы (если доступно). Есть ли у него вкладка или раздел с дополнительными характеристиками? Что сломается в работе пользователей, если удалить вид характеристики из плана видов характеристик — но не удалять уже накопленные значения в регистре?

**Спроектируй.** Вам нужно добавить в «Канцтовары» возможность хранить произвольные характеристики товаров. Как изменится дерево метаданных? Назовите объекты, которые нужно добавить, и опишите, как они связаны между собой.

### Ответы и указания

*Вопрос 5.* Если фирма торгует разными группами товаров (канцелярия, электроника, одежда), у каждой группы свои характеристики. Добавить все возможные реквизиты в справочник «Товары» — значит создать огромное количество полей, большинство из которых будут пустыми для конкретного товара. ПВХ позволяет хранить только те характеристики, которые реально нужны каждому конкретному объекту.

*Упражнение «Спроектируй».* Добавить в дерево: (1) «ПВХ ВидыХарактеристикТоваров» — пользователь добавляет виды (Цвет, Размер, ...); (2) «Регистр сведений ХарактеристикиТоваров» — измерения: Товар (→Товары), ВидХарактеристики (→ПВХ ВидыХарактеристикТоваров), ресурс: Значение. Измерение «ВидХарактеристики» ссылается именно на ПВХ — это обеспечивает контроль допустимых видов: нельзя записать значение для характеристики, которой нет в плане.

### Мостик

ПВХ — инструмент для гибкого описания объектов учёта. Следующие объекты отвечают за другое: не за то, что хранится в базе, а за то, как организована работа людей с системой.
