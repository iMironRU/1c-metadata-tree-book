## § 6.2. Движения и регистраторы: смотрим с другой стороны

### Открытие

В Модулях 3, 4 и 5 мы смотрели на связь «документ → регистр» со стороны документа: открывали вкладку «Движения» в его настройках и видели, в какой регистр он пишет. Теперь давайте посмотрим с другой стороны: от регистра — назад к документам.

Откройте в Конфигураторе регистр накопления «ОстаткиТоваров». Перейдите на вкладку **«Регистраторы»**.

### Что показывает вкладка «Регистраторы»

На вкладке «Регистраторы» видно, какие документы настроены как источники движений для этого регистра. Для «ОстаткиТоваров» в «Канцтоварах» там один документ — «ПоступлениеТоваров».

В реальных конфигурациях список бывает длинным:

```
Регистр накопления «ОстаткиТоваров»
│
└── Регистраторы:
    ├── ПоступлениеТоваров     (Приход  — товары приходят на склад)
    ├── РеализацияТоваров      (Расход  — товары уходят покупателю)
    ├── ПеремещениеТоваров     (Расход + Приход — с одного склада на другой)
    └── СписаниеТоваров        (Расход  — товары признаны негодными)
```

Каждый из этих документов при проведении создаёт движения в регистре. Регистр один, документов — несколько.

> Список регистраторов — это список бизнес-событий, которые меняют состояние учётной единицы. Если в регистр «ОстаткиТоваров» пишут четыре документа — значит, в системе есть четыре вида событий, влияющих на складские остатки.

### Один документ — несколько регистров

Посмотрим с другой стороны: один документ может писать в несколько регистров одновременно. Например, документ «РеализацияТоваров» в развитой конфигурации пишет:

```
Документ «РеализацияТоваров»
│
└── Движения в регистры:
    ├── ОстаткиТоваров             (Расход — товар ушёл со склада)
    ├── ВзаиморасчётыСКонтрагентами (Приход — покупатель стал должен деньги)
    └── ПродажиЗаПериод            (Оборот — факт продажи для отчётности)
```

Одно бизнес-событие — несколько учётных последствий. Документ фиксирует событие один раз; регистры фиксируют каждое из его последствий отдельно.

### Как использовать это при чтении чужой конфигурации

Список регистраторов — это быстрая карта бизнес-процессов, связанных с данным регистром. Алгоритм чтения от регистра:

```
Шаг 1. Откройте регистр → вкладка «Регистраторы»
        Кто пишет в этот регистр?
           ↓
Шаг 2. Откройте один из документов-регистраторов
        Посмотрите его реквизиты — на какие справочники ссылается?
           ↓
Шаг 3. Откройте вкладку «Движения» документа
        В какие ещё регистры он пишет?
           ↓
Шаг 4. Сформулируйте цепочку вслух:
        Справочники → Документ → Регистры → Отчёты
```

Пример для «Канцтоваров»:

```
Регистр «ОстаткиТоваров»
  ↓ Регистраторы: ПоступлениеТоваров
Документ «ПоступлениеТоваров»
  ↓ Реквизиты: Контрагент (→Контрагенты), Склад (→Склады)
  ↓ Табличная часть: Товар (→Товары)
Справочники «Товары», «Контрагенты», «Склады»
```

Связь между документом и регистром читается с двух сторон:

```
Со стороны регистра:                 Со стороны документа:
Регистр «ОстаткиТоваров»             Документ «РеализацияТоваров»
  └── Регистраторы:                    └── Движения:
      ├── ПоступлениеТоваров               ├── ОстаткиТоваров
      ├── РеализацияТоваров               └── ВзаиморасчётыСКонтрагентами
      └── ...
```

Открываете регистр → видите, какие события его меняют. Открываете документ → видите, какие состояния он затрагивает.

### Движения от разных документов: вид движения

Разные документы пишут в один регистр с разным видом движения. «ПоступлениеТоваров» — Приход, «РеализацияТоваров» — Расход. Именно поэтому остаток считается правильно: остаток = сумма приходов − сумма расходов по каждому сочетанию измерений.

Платформа суммирует числа. Смысл задаёт разработчик, правильно указав вид движения при настройке каждого документа.

### Главное

Связь «документ → регистр» видна с двух сторон: со стороны документа (вкладка «Движения») и со стороны регистра (вкладка «Регистраторы»). Один регистр обслуживается несколькими документами — это значит, несколько бизнес-событий влияют на одну учётную единицу. Один документ пишет в несколько регистров — одно событие порождает несколько учётных последствий. Список регистраторов — это быстрая карта того, какие события в системе меняют данный учётный показатель.

### Контрольные вопросы

1. Где в Конфигураторе можно увидеть список всех документов, пишущих в конкретный регистр?
2. Что означает, что у регистра «ОстаткиТоваров» четыре документа-регистратора?
3. Почему один документ пишет в несколько регистров, а не в один общий?
4. Чем отличается вид движения «Приход» от «Расход» с точки зрения расчёта остатка?
5. Если добавить в конфигурацию новый документ, который должен уменьшать остаток товаров, — какие настройки нужно сделать?

### Упражнения

**Найди в Конфигураторе.** Откройте демо-базу. Найдите любой регистр накопления и перейдите на вкладку «Регистраторы». Сколько документов в списке? Откройте один из них и перейдите на его вкладку «Движения» — есть ли другие регистры, кроме исходного? Что это говорит о бизнес-событии, которое фиксирует этот документ?

**Посмотри и ответь.** В демо-базе найдите документ, который пишет одновременно в несколько регистров. Откройте вкладку «Движения» — сколько регистров в списке? Что произойдёт с данными в регистре, если убрать этот документ из числа его регистраторов (не удалять документ, а убрать настройку движений)? Изменятся ли уже проведённые документы или только поведение будущих?

**Спроектируй.** В «Канцтовары» решили добавить документ «РеализацияТоваров» (продажа товара покупателю). В какие регистры он должен писать и с каким видом движения? Нарисуйте ASCII-схему: документ → регистры → вид движения в каждом.

### Ответы и указания

*Вопрос 3.* Каждый регистр отвечает за свой вид данных — остатки отдельно, взаиморасчёты отдельно, продажи отдельно. Один общий регистр превратился бы в структуру с несовместимыми полями и запутанными данными.

*Вопрос 5.* Нужно: (1) создать документ; (2) на вкладке «Движения» добавить регистр «ОстаткиТоваров»; (3) указать вид движения «Расход»; (4) настроить соответствие полей документа полям регистра через Конструктор движений.

*Упражнение «Спроектируй».*

```
РеализацияТоваров
├── → ОстаткиТоваров             (Расход — товар ушёл со склада)
└── → ВзаиморасчётыСКонтрагентами (Приход — покупатель стал должен)
```

Если в конфигурации ещё нет регистра взаиморасчётов — его нужно создать вместе с документом.

### Мостик

Теперь мы умеем читать связи в обоих направлениях: от документа к регистру и от регистра к документу. Но в реальных конфигурациях некоторые реквизиты появляются сразу во многих объектах — они вынесены на уровень всей конфигурации. Посмотрим на эту конструкцию.
