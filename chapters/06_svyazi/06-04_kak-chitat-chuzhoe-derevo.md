## § 6.4. Как читать чужое дерево: три стратегии навигации

### Открытие

Вы знаете все основные объекты и все виды связей между ними. Теперь — главный навык Модуля 6: взять незнакомое дерево и прочитать его как описание бизнес-процессов, а не как список объектов.

Откройте Конфигуратор с демо-базой (не «Канцтовары», а готовую конфигурацию). Перед вами — сотни объектов. С чего начать?

### Почему нельзя читать сверху вниз

Дерево метаданных упорядочено по типам объектов, а не по бизнес-логике. Сначала — все справочники, потом — все документы, потом — все регистры. Если читать сверху вниз, вы получите длинный список без связей: «Товары... Контрагенты... Склады... ПоступлениеТоваров... РеализацияТоваров...» — и непонятно, как они соотносятся друг с другом.

> Бизнес-логика проходит поперёк дерева: справочник → документ → регистр → отчёт. Дерево организовано по типам объектов, а читать его нужно по бизнес-цепочкам.

### Стратегия 1: от отчёта к справочнику

Отчёт — это конец цепочки: он задаёт вопрос к данным. По отчёту можно восстановить всю цепочку в обратном направлении.

```
Шаг 1. Откройте любой отчёт в дереве.
        Откройте его схему компоновки данных (СКД — настройки отчёта из Модуля 5).
        Найдите источник данных → какой регистр используется?
           ↓
Шаг 2. Откройте этот регистр.
        Перейдите на вкладку «Регистраторы».
        Какие документы пишут в него?
           ↓
Шаг 3. Откройте один из документов.
        Посмотрите реквизиты → на какие справочники он ссылается?
           ↓
Шаг 4. Сформулируйте цепочку вслух:
        Справочники → Документ → Регистр → Отчёт
```

Пример для «Канцтоваров»:

```
Отчёт «ОстаткиТоваровНаСкладе»
  ↓ СКД → источник: ОстаткиТоваров.Остатки
Регистр накопления «ОстаткиТоваров»
  ↓ Регистраторы: ПоступлениеТоваров
Документ «ПоступлениеТоваров»
  ↓ Реквизиты: Контрагент (→Контрагенты), Склад (→Склады)
  ↓ Табличная часть: Товар (→Товары)
Справочники «Товары», «Контрагенты», «Склады»
```

Эта стратегия удобна, когда нужно понять, откуда берутся данные в конкретном отчёте.

### Стратегия 2: от документа к данным

Если известно бизнес-событие, начинаем с документа, который его фиксирует:

```
Шаг 1. Откройте документ в дереве.
        Посмотрите реквизиты → что и на что ссылается?
           ↓
Шаг 2. Перейдите на вкладку «Движения».
        В какие регистры пишет этот документ?
           ↓
Шаг 3. Откройте каждый из регистров.
        Какой вид учёта он ведёт — остатки, сведения, обороты?
           ↓
Шаг 4. Найдите отчёты, которые используют эти регистры.
```

Эта стратегия удобна, когда нужно понять конкретную операцию: «что происходит в системе при оформлении возврата?» или «что меняется при проведении документа X?».

### Стратегия 3: от справочника к использованию

Если известен объект учёта — товар, контрагент, проект — начинаем с него:

```
Шаг 1. Найдите справочник в дереве.
        Посмотрите его реквизиты и табличные части.
           ↓
Шаг 2. Найдите документы, которые ссылаются на этот справочник.
        (пока проще делать это просмотром документов по очереди, проверяя
        типы их реквизитов; инструменты поиска по метаданным разберём отдельно)
           ↓
Шаг 3. Через каждый документ — к регистрам,
        через регистры — к отчётам.
           ↓
Шаг 4. Сформулируйте вслух:
        «Справочник X участвует в таких-то операциях
         и влияет на такие-то показатели»
```

Эта стратегия удобна, когда нужно понять полный жизненный цикл сущности: как товар появляется в системе, как движется, где и как анализируется.

### Как записывать то, что читаешь

При чтении незнакомого дерева полезно зарисовывать цепочки на бумаге — не весь граф, а ключевые связи для конкретных бизнес-процессов:

```
[Товары] ─────────────────────────────────────┐
[Контрагенты] ──→ [ПоступлениеТоваров] ──→ [ОстаткиТоваров] ──→ [Отчёт]
[Склады] ──────────────────────────────────────┘
```

Такой рисунок помогает увидеть, чего не хватает: нет расходного документа — значит, остатки только растут; есть регистр, но нет отчёта — данные накапливаются, но никак не анализируются.

### Три ключевых вопроса для любого объекта

Независимо от стратегии, для каждого незнакомого объекта задавайте три вопроса:

1. **Что здесь хранится?** (справочник — классификатор, регистр — состояние или история)
2. **Что здесь фиксируется?** (документ — бизнес-событие)
3. **Что здесь можно спросить?** (отчёт — вопрос к накопленным данным)

Ответив на эти три вопроса для каждого объекта в цепочке, вы получите связный бизнес-процесс.

### Главное

Чужое дерево читается не сверху вниз, а по бизнес-смыслу — через цепочки объектов. Три стратегии навигации: от отчёта к справочнику (обратная цепочка), от документа к данным (прямая цепочка), от справочника к использованию (полный жизненный цикл). В каждой стратегии центральный вопрос — «какую бизнес-задачу решает этот объект и с кем он связан?». Зарисовывайте цепочки — они помогают увидеть пробелы в логике.

### Контрольные вопросы

1. Почему чужое дерево трудно читать сверху вниз, от первого объекта к последнему?
2. Какую стратегию удобнее использовать, если нужно понять, откуда берутся данные в конкретном отчёте?
3. Как найти все документы, которые ссылаются на справочник «Товары», не открывая каждый объект по отдельности?
4. Что делать, если в регистре есть регистраторы, но нет ни одного отчёта, обращающегося к нему?
5. Как понять, что в конфигурации есть «незакрытый» бизнес-процесс — операция без документа или документ без регистра?

### Упражнения

**Найди в Конфигураторе (упражнение на цепочку).** Откройте демо-базу. Возьмите любой отчёт из ветки «Отчёты». Откройте его СКД → вкладка «Наборы данных» → найдите источник данных. Определите регистр. Откройте регистр → «Регистраторы» → найдите документы. Откройте один документ → посмотрите реквизиты шапки. Напишите полную цепочку: Справочники → Документ → Регистр → Отчёт.

**Посмотри и ответь.** Откройте демо-базу и найдите регистр накопления, у которого в списке регистраторов есть только документы с видом движения «Приход» — без «Расхода». Что это означает для бизнес-процесса? Как накапливаются данные в регистре, если расходных движений нет? Является ли это проблемой или нормальной ситуацией для некоторых задач?

**Спроектируй (мини-анализ).** Вам дано следующее дерево метаданных:
```
Справочники:    Сотрудники, Должности
Документы:      НачислениеЗарплаты, ВыплатаЗарплаты
Регистры накопления: ЗарплатаКВыплате
Регистры сведений:   ОкладыСотрудников
Отчёты:         ЗарплатнаяВедомость
```
Восстановите вероятные связи между объектами: какой документ пишет в какой регистр, какой регистр использует отчёт, на какие справочники ссылаются документы. Опишите бизнес-процесс одним абзацем.

### Ответы и указания

*Вопрос 3.* Точного автоматического поиска в стандартном Конфигураторе нет. Практический метод: использовать поиск по метаданным («Правка» → «Найти» → ввести «СправочникСсылка.Товары» в поиске по реквизитам) или просматривать документы по очереди, проверяя типы их реквизитов.

*Вопрос 4.* Регистр без отчётов — данные накапливаются, но не анализируются через стандартный интерфейс. Возможные причины: отчёт подключён внешним файлом; данные используются во внутренней логике при проведении других документов; отчёт ещё не создан — незавершённая разработка.

*Упражнение «Спроектируй».* Вероятная схема: «НачислениеЗарплаты» пишет в «ЗарплатаКВыплате» (Приход), ссылается на «Сотрудники» и «Должности»; «ВыплатаЗарплаты» пишет в «ЗарплатаКВыплате» (Расход), ссылается на «Сотрудники»; «ОкладыСотрудников» хранит текущий оклад (измерение: Сотрудник, ресурс: Оклад); «ЗарплатнаяВедомость» использует «ЗарплатаКВыплате.Остатки». Бизнес-процесс: «В системе хранятся сотрудники и их должности. Ежемесячно начисляется зарплата — документ создаёт долг перед сотрудником. После выплаты оформляется второй документ — долг погашается. Отчёт показывает, кому и сколько ещё должны выплатить.»

### Мостик

Теперь вы умеете читать чужое дерево по стратегиям. Осталось применить это к знакомому дереву — «Канцтовары» — и прочитать его целиком как единый бизнес-процесс, от первого справочника до последнего отчёта.
