## § 3.6. Понятие регистратора: документ берёт ответственность

### Открытие

В §3.5 мы разобрали *зачем* нужен документ-регистратор — чтобы знать причину изменения. Теперь разберём *как* это устроено в Конфигураторе.

Откройте одновременно: вкладку «Движения» у документа «ПриказОбИзмenenИЦен» и вкладку «Основные» у регистра «ЦеныТоваров».

```
Документы → ПриказОбИзмenenИЦен → вкладка «Движения»
└── Регистр сведений «ЦеныТоваров»   ← здесь перечислены регистры,
                                        в которые документ пишет

Регистры сведений → ЦеныТоваров → вкладка «Основные»
└── Режим записи: Подчинённый регистратору   ← здесь указан режим
```

Связь прописана в двух местах. Одного без другого недостаточно: если у регистра стоит «Подчинённый», но ни один документ не указан как его регистратор — в регистр никто ничего не запишет.

### Механика ответственности

Регистратор — это документ, который при проведении создаёт записи в регистре. Ответственность выражается так:

**При проведении** — записи создаются.

**При отмене проведения** — платформа, как правило, удаляет записи, созданные этим документом, и при повторном проведении создаёт их заново из актуального содержимого документа. Регистр всегда отражает то, что написано в проведённом документе.

**При удалении документа** — его записи в регистре также перестают существовать.

> Регистратор — это документ, который несёт ответственность за записи в регистре. Записи существуют ровно до тех пор, пока документ существует и проведён.

### Один документ — несколько регистров

Один документ может быть регистратором сразу нескольких регистров. Когда в Модуле 4 мы создадим регистр накопления «ОстаткиТоваров», документ «ПоступлениеТоваров» будет писать туда при проведении. Тот же документ может одновременно писать в регистр сведений (например, фиксировать дату последней поставки) и в регистр накопления (фиксировать прирост остатков).

```
Документ «ПоступлениеТоваров»
└── вкладка «Движения»
    ├── Регистр сведений «ДатыПоследнихПоставок»  ← пишет при проведении
    └── Регистр накопления «ОстаткиТоваров»        ← тоже пишет (Модуль 4)
```

### Как документ знает, что писать

Когда документ указан как регистратор — платформа знает *куда* писать. Но *что именно* переносить из полей документа в поля регистра — это задаётся в настройках соответствия. В Конфигураторе для этого есть Конструктор движений: он позволяет указать без углубления в программирование, что «поле "Товар" из табличной части документа → измерение "Товар" в регистре, поле "Цена" → ресурс "Цена"». Подробности этой настройки — в практическом §3.7.

### Главное

Регистратор — это документ, который несёт ответственность за записи в регистре. Проведён — записи есть. Не проведён или удалён — записей нет. Один документ может быть регистратором нескольких регистров. Связь прописывается в двух местах: в вкладке «Движения» документа и в режиме записи регистра.

### Контрольные вопросы

1. Что происходит с записями в регистре, если отменить проведение документа-регистратора?
2. Почему связь между документом и регистром прописывается в двух местах?
3. Может ли один документ быть регистратором для нескольких регистров?
4. Может ли один регистр иметь несколько документов-регистраторов?
5. Почему разработчик не может просто написать данные в подчинённый регистр в обход документа — с точки зрения бизнес-логики?

### Упражнения

**Найди в Конфигураторе.** Откройте несколько документов в демо-базе. Проверьте вкладку «Движения» у каждого. Найдите документ, который является регистратором для более чем одного регистра. Выпишите: какие именно регистры он затрагивает?

**Посмотри и ответь.** Откройте подчинённый регистр сведений. Найдите в пользовательском режиме несколько записей и посмотрите на колонку «Регистратор». Откройте один из документов-регистраторов. Соответствует ли содержимое документа записям в регистре? Что будет, если исправить документ и перепровести — изменятся ли записи в регистре?

**Спроектируй.** Для «Канцтоваров» нужно: (а) хранить историю изменения цен с указанием документа-основания; (б) при проведении документа автоматически обновлять дату последней отгрузки для каждого контрагента. Для каждого придумайте: какой документ будет регистратором, в какой регистр он пишет, какие измерения и ресурсы в этом регистре.

### Ответы и указания

*Вопрос 4.* Да. Один и тот же регистр «ЦеныТоваров» может принимать записи и от «ПриказаОбИзмененииЦен», и от «ПоступленияТоваров» (если при поступлении автоматически фиксируется новая закупочная цена). В колонке «Регистратор» будут разные документы.

*Упражнение «Спроектируй» (б).* Документ «РеализацияТоваров» (или «ПоступлениеТоваров») — регистратор. Регистр сведений «ДатыПоследнихОтгрузок», режим «Подчинённый». Измерение: «Контрагент». Ресурс: «ДатаОтгрузки» (тип Дата). При проведении каждого документа реализации — запись обновляется.
