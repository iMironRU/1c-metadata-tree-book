## § 3.7. Создаём регистр «ЦеныТоваров» и документ «ПриказОбИзмененииЦен»

### Открытие

В §1.5 мы задавали вопрос: почему плохо хранить цену как реквизит справочника «Товары»? Потому что теряется история. Теперь мы знаем решение: регистр сведений с периодичностью.

Создадим регистр «ЦеныТоваров» и документ «ПриказОбИзмененииЦен», который будет в него писать.

### Шаг 1: создаём регистр сведений «ЦеныТоваров»

В дереве метаданных найдите ветку «Регистры сведений». Правой кнопкой → «Добавить».

**Вкладка «Основные»:**

```
Имя:             ЦеныТоваров
Синоним:         Цены товаров
Режим записи:    Подчинённый регистратору
Периодичность:   По позиции регистратора
```

«По позиции регистратора» означает: период не фиксированный (не «По дням»), а привязан к конкретной записи документа. Одна позиция документа = одна запись в регистре с конкретным моментом.

**Вкладка «Данные»:**

В разделе «Измерения» добавьте:

```
Товар   — тип: СправочникСсылка.Товары
```

В разделе «Ресурсы» добавьте:

```
Цена    — тип: Число, точность 2
```

Нажмите ОК.

```
Регистр сведений «ЦеныТоваров»
├── Режим записи:    Подчинённый регистратору
├── Периодичность:   По позиции регистратора
├── Измерения:
│   └── Товар   (СправочникСсылка.Товары)
└── Ресурсы:
    └── Цена    (Число)
```

### Шаг 2: создаём документ «ПриказОбИзмененииЦен»

В ветке «Документы» → правой кнопкой → «Добавить».

**Вкладка «Основные»:**

```
Имя: ПриказОбИзмenenИЦен
```

**Вкладка «Данные»:**

Реквизиты шапки не добавляем — пока хватит стандартных «Номер» и «Дата».

Добавьте табличную часть «Товары» с реквизитами:

```
Товар   — тип: СправочникСсылка.Товары
Цена    — тип: Число, точность 2
```

**Вкладка «Движения»:**

Нажмите «Добавить» и выберите регистр сведений «ЦеныТоваров». Нажмите ОК.

```
Документ «ПриказОбИзмененииЦен»
├── Реквизиты: Номер, Дата
├── Табличная часть «Товары»:
│   ├── Товар   (СправочникСсылка.Товары)
│   └── Цена    (Число)
└── Движения:
    └── Регистр сведений «ЦеныТоваров»
```

### Шаг 3: настроить соответствие полей

Вкладка «Движения» в Конфигураторе позволяет указать, *куда* документ пишет. Но *что именно* он переносит из своих полей в поля регистра — задаётся отдельно.

Прежде чем идти дальше — важная оговорка. Сейчас мы впервые используем механизм, который устанавливает это соответствие автоматически. Понимать его устройство изнутри не требуется — это уровень программирования, выходящий за рамки нашего курса. Наша задача — увидеть саму идею: документу нужно объяснить, какие его поля становятся какими полями регистра.

Для этого в Конфигураторе есть **Конструктор движений**. Нажмите на кнопку «Конструктор движений» на вкладке «Движения». Откроется диалог, где вы укажете соответствие:

```
Поле документа «Товар» (из табличной части)  →  Измерение «Товар» в регистре
Поле документа «Цена» (из табличной части)   →  Ресурс «Цена» в регистре
```

Нажмите ОК. Платформа создаст необходимые настройки — документ теперь знает, что именно переносить в регистр при проведении.

### Шаг 4: сохранить и проверить

Нажмите F7 (сохранить конфигурацию), F8 (обновить базу).

В пользовательском режиме:
- Создайте новый «ПриказОбИзмененииЦен».
- Добавьте строки: выберите товар из справочника, укажите новую цену.
- Нажмите «Провести».
- Откройте регистр сведений «ЦеныТоваров» — там должны появиться строки.

### Итоговая схема

```
Конфигурация «Канцтовары»
│
├── Справочники
│   ├── Товары
│   └── Контрагенты
│
├── Документы
│   ├── ПоступлениеТоваров
│   └── ПриказОбИзмenenИЦен     ← регистратор
│       └── Движения: ЦеныТоваров
│
└── Регистры сведений
    └── ЦеныТоваров              ← подчинённый регистратору
        ├── Измерения: Товар
        └── Ресурсы: Цена
```

### Главное

Регистр сведений «ЦеныТоваров» хранит историю цен: для каждого товара — цена с указанием момента изменения. Данные попадают в регистр через документ «ПриказОбИзмененииЦен» при проведении. Если документ отменить — цены из этого приказа исчезнут из регистра. Теперь у «Канцтоваров» есть полноценная история цен с привязкой к документу-основанию.

### Контрольные вопросы

1. Зачем для хранения цен нужен регистр — почему не обойтись реквизитом справочника «Товары»?
2. Что означает периодичность «По позиции регистратора»?
3. Что нужно сделать в Конфигураторе, чтобы документ стал регистратором для регистра?
4. Как проверить, что документ правильно заполнил регистр?
5. Что произойдёт, если создать два приказа об изменении цен для одного товара?

### Упражнения

**Сделай в Конфигураторе.** Создайте регистр «ЦеныТоваров» и документ «ПриказОбИзмenenИЦен» по шагам выше. Проведите два документа с изменением цены одного товара. Откройте регистр — сколько строк появилось? Как отсортированы записи?

**Посмотри и ответь.** Откройте регистр «ЦеныТоваров» в пользовательском режиме. Найдите строки с одним товаром. Нажмите на значение в колонке «Регистратор» — откроется документ. Теперь отмените проведение этого документа. Снова откройте регистр — что изменилось?

**Спроектируй.** Добавьте в регистр «ЦеныТоваров» второе измерение — «ТипЦены» (Розничная / Оптовая). Что изменится в структуре таблицы? Что нужно изменить в документе «ПриказОбИзмenenИЦен», чтобы он мог устанавливать разные цены для разных типов?

### Ответы и указания

*Вопрос 5.* Оба приказа создадут записи в регистре. Если даты у приказов разные — в регистре будет два значения для этого товара в разные периоды. Система всегда сможет дать «актуальную» цену: ближайшую к запрашиваемой дате. Если даты совпадают — второй приказ создаст вторую запись на ту же дату; поведение системы зависит от настроек (обычно берётся последняя запись по времени).

*Упражнение «Спроектируй».* В табличной части документа нужен дополнительный реквизит «ТипЦены» (например, перечисление). В настройках соответствия Конструктора движений его значение нужно связать с измерением регистра. Теперь одна строка документа — одна запись в регистре с двумя ключами: Товар + ТипЦены.
