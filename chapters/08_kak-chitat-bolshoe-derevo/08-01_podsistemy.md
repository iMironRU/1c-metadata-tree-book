# Модуль 8. Как читать большое незнакомое дерево

## § 8.1. Подсистемы: первый шаг перед любой навигацией

### Открытие

В Модулях 1–7 мы изучали объекты по одному и работали с «Канцтоварами» — конфигурацией из восьми объектов, где всё на виду. Теперь откройте демо-базу типовой конфигурации — «Управление торговлей», «Бухгалтерия предприятия» или любую другую промышленную конфигурацию.

Вы увидите несколько сотен объектов. Справочников — пятьдесят. Документов — сто. Регистров — ещё полсотни. Все они перечислены в алфавитном порядке внутри своего типа.

С чего начать? Сверху вниз — не работает.

### Почему нельзя читать большое дерево сверху вниз

Дерево метаданных упорядочено по **типам объектов**, а не по бизнес-логике. Все справочники идут вместе, все документы — вместе, все регистры — вместе. Если читать по порядку, вы получите бессвязный поток имён без понимания того, как они соотносятся друг с другом.

Реальная бизнес-логика проходит **поперёк** дерева — от справочника через документ к регистру и отчёту. Но это вы уже знаете из Модуля 6.

Новая проблема большого дерева: в нём не одна такая цепочка, а десятки. И прежде чем читать отдельные цепочки, нужно понять, на какие **крупные блоки** делится конфигурация.

### Подсистема — это кластер объектов вокруг одной бизнес-задачи

Любая крупная конфигурация состоит из подсистем. Подсистема — это группа объектов, обслуживающих одну бизнес-задачу: закупки, продажи, склад, взаиморасчёты, зарплата.

```
Типовая торговая конфигурация — крупные подсистемы:

┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│    Закупки      │  │    Продажи      │  │      Склад      │
│  Поставщики     │  │  Покупатели     │  │  Номенклатура   │
│  Поступления    │  │  Реализации     │  │  Перемещения    │
│  Взаиморасч.   │  │  Взаиморасч.   │  │  Остатки        │
└─────────────────┘  └─────────────────┘  └─────────────────┘

┌─────────────────┐  ┌─────────────────┐
│  Ден. средства  │  │    Зарплата     │
│  Платёжки       │  │  Сотрудники     │
│  Кассовые ордера│  │  Начисления     │
└─────────────────┘  └─────────────────┘
```

Подсистемы видны в дереве как кластеры объектов с похожими именами. Они не оформлены отдельной веткой — их нужно опознать по именованию и связям.

### Как найти подсистемы: три сигнала

**Сигнал 1: именование группами.** Объекты одной подсистемы обычно содержат общие слова. Если в дереве много документов с «Закупка» или «Поставщик» в имени — это кластер. Поищите такие слова среди справочников, документов и регистров одновременно.

**Сигнал 2: регистры накопления как «центры тяжести».** Каждый регистр накопления — это учётный показатель. Он обычно обслуживает одну подсистему. Посмотрите на список регистров накопления — их имена часто прямо называют подсистему: «ОстаткиТоваров», «ВзаиморасчётыСКонтрагентами», «ДвиженияДенежныхСредств».

**Сигнал 3: отчёты как итог подсистемы.** Отчёт всегда принадлежит какой-то подсистеме — он отвечает на её вопросы. Пройдитесь по списку отчётов и сгруппируйте их по теме: «вот отчёты по продажам, вот по складу, вот по деньгам».

### Первый шаг при чтении нового дерева

```
Шаг 1. Посмотрите список регистров накопления
        (их немного, имена говорящие)
        → составьте список подсистем
           ↓
Шаг 2. Для каждой подсистемы найдите 1–2 отчёта
        → они покажут, на какие вопросы отвечает подсистема
           ↓
Шаг 3. Найдите ключевые документы каждой подсистемы
        → это «глаголы», вокруг которых строится логика
           ↓
Шаг 4. Только теперь читайте отдельные цепочки
```

Не пытайтесь охватить всё сразу. Выберите одну подсистему — и читайте её полностью. Потом следующую.

> Большое дерево читается не как список, а как карта районов города: сначала нужно понять, где находятся «центры», а потом исследовать каждый район отдельно.

### Главное

Большое дерево нельзя читать сверху вниз — объекты сгруппированы по типам, а не по бизнес-логике. Первый шаг: найти подсистемы — крупные кластеры объектов вокруг одной бизнес-задачи. Их можно опознать по трём сигналам: именование группами, регистры накопления как «центры тяжести», отчёты как итог подсистемы. Только найдя подсистемы, переходите к чтению отдельных цепочек.

### Контрольные вопросы

1. Почему поиск подсистем — это первый шаг, а не сразу чтение отдельных объектов?
2. Почему регистры накопления удобно использовать для поиска подсистем?
3. В конфигурации 60 документов. Как быстро найти, какие из них относятся к подсистеме продаж?
4. Что делать, если у объекта имя не указывает ни на одну из очевидных подсистем?
5. Можно ли в одной конфигурации найти объекты, которые относятся сразу к двум подсистемам? Приведите пример.

### Упражнения

**Найди в Конфигураторе.** Откройте демо-базу типовой конфигурации. Откройте ветку «Регистры накопления». Перечислите все регистры и попробуйте сгруппировать их в подсистемы только по именам. Сколько подсистем вы насчитали? Совпадает ли это с вашим ожиданием от торговой или бухгалтерской системы?

**Посмотри и ответь.** Откройте ветку «Отчёты» в той же конфигурации. Сгруппируйте отчёты по подсистемам — снова только по именам. Есть ли отчёты, которые трудно отнести к конкретной подсистеме? Что это говорит об их назначении — они кросс-подсистемные или просто плохо названы?

**Спроектируй.** Вам дана конфигурация для управления рестораном. Назовите 4–5 вероятных подсистем. Для каждой — один регистр накопления и один документ, которые вы ожидали бы в ней найти.

### Ответы и указания

*Вопрос 2.* Регистров накопления в конфигурации обычно немного (в отличие от документов или справочников), и их имена напрямую отражают учётный показатель. «ОстаткиТоваров» — складская подсистема, «РасчётыСПоставщиками» — закупки, «ДвиженияДенежныхСредств» — казначейство. Это быстрый способ получить карту подсистем без чтения каждого объекта.

*Вопрос 5.* Да. Справочник «Контрагенты» используется и в закупках, и в продажах, и во взаиморасчётах. Некоторые регистры тоже могут обслуживать несколько подсистем. Такие «общие» объекты — признак того, что они являются ключевыми классификаторами системы.

*Упражнение «Спроектируй».* Вероятные подсистемы ресторана: (1) Меню и номенклатура — справочник блюд, регистр состава блюд; (2) Заказы — документ «ЗаказСтола», регистр «ВыручкаПоБлюдам»; (3) Склад ингредиентов — документ «ПоступлениеПродуктов», регистр «ОстаткиПродуктов»; (4) Кассовые операции — документ «Чек», регистр «ДвиженияДенег»; (5) Персонал — документ «НачислениеЗарплаты», регистр «ЗарплатаКВыплате».

### Мостик

Вы нашли подсистемы и выбрали одну для чтения. Теперь — первый конкретный приём: читаем цепочку в обратном направлении, начиная с отчёта.
