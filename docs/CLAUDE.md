# CLAUDE.md — Книга «Дерево метаданных 1С: как читать и понимать»

Этот файл — точка восстановления для продолжения работы над книгой.

---

## Что за проект

Учебная книга по дереву метаданных 1С для новичков. Цель читателя: научиться читать чужое дерево метаданных и переводить его на бизнес-язык. Не программировать — именно читать и понимать.

Сквозной проект: фирма **«Канцтовары»** — торговля канцелярскими товарами. В каждом практическом параграфе читатель добавляет новый объект в конфигурацию «Канцтовары».

## Структура каждого параграфа (строго соблюдать)

1. **Открытие** — связь с предыдущим + где смотреть в Конфигураторе
2. **Содержательные разделы** (4–7) с выносками `>`, примерами, ASCII-схемами
3. **Главное** — один плотный абзац
4. **Контрольные вопросы** — 5 штук
5. **Упражнения** — минимум три вида:
   - «Найди в Конфигураторе» (наблюдение)
   - «Посмотри и ответь» (интерпретация: **обязательно «что сломается, если изменить?»**)
   - «Спроектируй» (генерация)
6. **Ответы и указания**
7. **Мостик** к следующему параграфу

## Правила письма (якорные, нарушать нельзя)

### Стиль
- **Конфликт перед объяснением**: сначала — что ломается без объекта, потом — решение
- **Одна метафора — один объект**:
  - «Язык» (существительные/глаголы/прилагательные) — рамка всего курса
  - «Амбарная книга» — только регистр накопления
  - «Прилагательное во времени» — только регистр сведений
  - «Имя собственное» — только ссылочный тип
- **Два уровня строгости**: разговорный текст в основе, строгое определение — только в выноске `>`
- **Никаких абсолютов**: не «не хранится нигде», а «не предназначен для»; не «удаляются навсегда», а «как правило, удаляются платформой»

### Терминология регистров (строго)
- **запись** — логическая единица регистра (набор полей)
- **строка** — только визуальное представление в пользовательском режиме
- **значение** — содержимое конкретного поля
- **состояние** — бизнес-смысл (цена, курс)

### Что запрещено
- Слова **«код», «модуль», «генерация кода»** — даже в контексте «это делается автоматически»
  - Вместо этого: «настройки соответствия», «Конструктор движений», «платформа создаёт настройки»
- Справочник API 1С — не пишем
- Эмодзи, цветные блоки, кричащая вёрстка

## Сквозные элементы, которые нужно поддерживать

### Формула записи регистра (введена в Модуле 3, §3.3)

Повторяется во всех параграфах про регистры:
```
Измерения  → о чём запись (ключ)
Ресурсы    → что известно (значение)
Период     → когда это было верно
Регистратор → почему это появилось
```

### Сравнение объектов (введено в §3.3)
```
Справочник       → «что это такое?»
Документ         → «что произошло?»
Регистр сведений → «чему это равно прямо сейчас?»
Регистр накопления → «сколько накопилось?»   ← добавить в Модуле 4
```

## История правок редактора (паттерны)

1. **Смешение типов состояния** — не смешивать примеры регистра сведений (цена, курс) и регистра накопления (остатки) в одном параграфе
2. **Абсолютные формулировки** — всегда смягчать
3. **Упоминание кода** — запрещено
4. **Слабые упражнения «посмотри и ответь»** — должны содержать интерпретационный вопрос
5. **Дублирование параграфов** — жёстко разводить смысл

## Как продолжать работу

1. Прочитать этот файл
2. Прочитать `docs/plan.md` — раздел «Модуль N» перед написанием
3. Посмотреть итоговую схему предыдущего модуля (`chapters/0N.../0N-0M_itog.md`)
4. Писать модуль: создать папку `chapters/0N_name/`, файлы `0N-01_...md` → `0N-0M_itog.md`
5. После написания — ждать правок редактора, анализировать, вносить
