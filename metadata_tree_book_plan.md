# Дерево метаданных 1С: как читать и понимать

*Рабочий план книги. Исправленная редакция.*

---

## I. Концепция курса

### Главная идея

Книга рассматривает **дерево метаданных 1С** как чертёж деловой системы: набор объектов, каждый из которых отвечает на вопрос *«что именно мы хотим сказать о деловом мире?»*.

Метаданные — это не просто список галочек и свойств. Это **архитектура смысла**, заложенная автором конфигурации. Умение читать дерево метаданных — это умение видеть, как бизнес-сущности (товары, клиенты, заказы, оплаты) превращаются в объекты конфигурации (справочники, документы, регистры, отчёты).

Книга не учит программировать в 1С. Она учит **ориентироваться в структуре** любой конфигурации и **понимать замысел** её автора — чтобы потом создавать свою.

### Аудитория

**Абсолютный новичок без опыта программирования и без знакомства с 1С.**

Жёсткие правила:

- Нельзя опираться на интуицию из других языков или систем.
- Объяснять не только *что* делает объект, но и *почему* он устроен именно так.
- Каждый новый объект — концептуальный сдвиг.
- Медленный старт, ускорение к концу.

### Это практический курс

Читатель работает в Конфигураторе с самого начала. Каждый параграф предполагает, что 1С установлена и открыта. ASCII-схемы в тексте — это **навигационные карты**, а не замена живому дереву: читатель смотрит одновременно и на схему в книге, и на Конфигуратор на экране.

Упражнения делятся на два вида:
- **«Найди в Конфигураторе»** — навигация и наблюдение в живом дереве.
- **«Спроектируй»** — нарисовать или описать фрагмент дерева для новой задачи.

### Цель выхода

К концу книги читатель умеет:

- Открыть любую конфигурацию в Конфигураторе и **назвать** каждый встреченный тип объекта.
- **Объяснить** бизнес-задачу, которую решает конкретный объект.
- **Спроектировать** дерево метаданных для простой задачи с нуля.
- **Прочитать** чужое дерево связным текстом на русском языке — как описание бизнес-процесса.

### Формат

- Учебник в стиле линейного чтения, уважительный тон.
- MD с ASCII-схемами дерева.
- Параграф = академическая пара (1,5 часа).
- Сквозной пример — **фирма «Канцтовары»** (преемственность с курсом по коду).

## II. Сквозные несущие конструкции

### Метафора языка

| Уровень | В естественном языке | В дереве метаданных 1С |
|---|---|---|
| 1 | буквы | типы объектов (справочник, документ…) |
| 2 | слова | конкретные объекты («Товары», «Поступление») |
| 3 | грамматика | правила связей (подчинение, владелец, движения) |
| 4 | предложения | набор объектов, описывающий один бизнес-процесс |
| 5 | текст | вся конфигурация как описание деловой системы |
| 6 | авторский замысел | архитектурные решения создателя конфигурации |
| 7 | чтение вслух | перевод дерева обратно на русский бизнес-язык |

### Части речи в дереве метаданных

```
Примитивный тип     →  описание          (Строка, Число, Дата, Булево —
                                           смысл внутри самого значения)
Ссылочный тип       →  имя собственное   (за именем скрыт целый объект
                                           со всеми его свойствами)
Справочник          →  существительное   (кто? что? — сущность)
Реквизит            →  прилагательное    (какой? — свойство)
Табличная часть     →  составное прилагательное (список свойств)
Документ            →  глагол            (что произошло? — событие)
Документ-регистратор→  глагол, меняющий таблицу (событие, фиксирующее причину)
Регистр сведений    →  прилагательное во времени (история значений с причиной)
Регистр накопления  →  «сколько чего стало» (амбарная книга итогов)
Отчёт               →  ответ на вопрос пользователя
```

### Сквозной сюжет: фирма «Канцтовары»

От параграфа к параграфу читатель вместе с вымышленным главным бухгалтером фирмы «Канцтовары» проектирует дерево метаданных с нуля и тут же создаёт объекты в Конфигураторе. Сначала — справочники, потом — документы, потом — регистры, потом — отчёты. К концу книги читатель не просто знает объекты, а **прожил процесс их создания**.

## III. Методические принципы

### Дидактика

- **Главное в одном абзаце** — плотный пересказ ключевых мыслей параграфа.
- **Три уровня самопроверки**: ответы → указания → полные решения.
- **Три вида упражнений**:
  - «Найди в Конфигураторе» — навигация и наблюдение (recognition).
  - «Посмотри и ответь» — наблюдение плюс анализ: «почему здесь так? что сломается, если изменить?» (interpretation).
  - «Спроектируй» — самостоятельное создание структуры (generation).
- Каждый параграф содержит минимум одно упражнение каждого вида из первых двух; параграфы с практикой в Конфигураторе — обязательно «посмотри и ответь».

### Стилистические правила

- Никаких абсолютов «без X нет Y».
- Бытовой регистр + строгий термин: «папка (объект метаданных, ветка дерева)».
- Ключевые мысли — выносками `>`.
- **Два уровня строгости**: основной текст — максимально простой, как разговор; строгая формулировка — только в выноске. Пример: основной текст — «ты создал справочник — теперь на него могут ссылаться другие объекты»; выноска — «объект метаданных добавляет новый ссылочный тип в систему».
- **Одна метафора — один объект**: метафоры не смешиваются в одном абзаце и не конкурируют. «Язык» — рамка всего курса. «Амбарная книга» — только для регистра накопления. «Прилагательное во времени» — только для регистра сведений. «Имя собственное» — только для ссылочного типа.
- **Конфликт перед объяснением**: каждый новый объект вводится через проблему, которую он решает. Сначала — что ломается без него, потом — решение. Не «объяснение → пример», а «проблема → ошибка → решение».

### Структурные правила

- Каждый параграф открывается связкой с предыдущим.
- Термины-якоря вводятся с оговоркой «мимоходом, вернёмся в §…».
- ASCII-схемы — навигационные карты для одновременной работы с Конфигуратором.

Пример ASCII-схемы:

```
Дерево метаданных (фрагмент — «Канцтовары»)
│
├── Справочники
│   ├── Товары
│   │   └── Реквизиты: Цена, Артикул, Единица
│   └── Контрагенты
│       └── Реквизиты: ИНН, Телефон, Адрес
│
├── Документы
│   └── ПоступлениеТоваров
│       ├── Реквизиты: Номер, Дата, Контрагент
│       └── Табличная часть «Товары»: Товар, Количество, Цена
│
└── Регистры накопления
    └── ОстаткиТоваров
        ├── Измерения: Товар, Склад
        └── Ресурсы: Количество
```

### Что мы НЕ делаем

- Не превращаем книгу в справочник API 1С.
- Не пишем и не разбираем код модулей.
- Не используем эмодзи, кричащую вёрстку, цветные блоки.

## IV. Формат параграфа

1. **Открытие** — связь с предыдущим + ориентир: где сейчас смотреть в Конфигураторе.
2. **Содержательные разделы** (4–7 штук) с выносками, примерами, ASCII-схемами.
3. **Главное в одном абзаце**.
4. **Контрольные вопросы** (5–7).
5. **Упражнения** (4–6, из них минимум 2 — в Конфигураторе).
6. **Ответы, указания, разборы**.
7. **Мостик к следующему параграфу**.

## V. Структура книги

### Модуль 0. Что такое дерево метаданных и как открыть Конфигуратор

Читатель запускает 1С, открывает Конфигуратор и видит дерево первый раз. С этого момента Конфигуратор открыт на протяжении всего курса.

- **§ 0.1.** Метаданные — это не программа, а описание устройства
- **§ 0.2.** Дерево как форма организации: корень, ветки, листья
- **§ 0.3.** Первый запуск Конфигуратора: где находится дерево и как его открыть *(практика)*
- **§ 0.4.** Что хранится и что делает: два разных слоя в Конфигураторе

  В Конфигураторе у каждого объекта есть вкладка «Модуль». Там живёт программный код — то, что *делает* система: реагирует на действия пользователя, проверяет данные, пишет в регистры. Это другой слой, другой уровень работы.

  Дерево метаданных описывает то, что *хранится*: какие объекты существуют, как они устроены, как связаны. Именно с этим мы работаем в курсе.

  > Есть то, что хранится — дерево метаданных. Есть то, что делает — модули. Мы работаем с первым. Модули — следующий уровень, у него есть отдельные курсы и книги.

- **§ 0.5.** Фирма «Канцтовары»: что мы хотим автоматизировать и что будем создавать

*Щёлкает:* читатель видит живое дерево, понимает его общую форму и знает, что именно будет делать в Конфигураторе.

### Модуль 1. Существительные — справочники

- **§ 1.1.** Справочник как список чего-либо (товары, сотрудники, контрагенты)
- **§ 1.2.** Реквизиты — свойства элемента (прилагательные). У каждого реквизита есть тип: Строка, Число, Дата, Булево. Тип определяет, что можно хранить — сложить два числа можно, сложить имя с датой нельзя. *(практика: посмотреть и ответить — какой тип у реквизита «Дата рождения»? Почему не Строка?)*
- **§ 1.3.** Иерархия в справочнике: группы и элементы
- **§ 1.4.** Табличные части — несколько строк свойств у одного элемента
- **§ 1.5.** Зачем вообще нужна структура: проблема без справочника

  Микро-инсайт Модуля 1. Можно хранить название контрагента прямо в каждом месте, где он встречается: в одном документе написать «ООО Ромашка», в другом — тоже. Но тогда при смене названия придётся найти и исправить каждое вхождение вручную. Справочник решает это иначе: контрагент хранится **один раз**, все остальные места на него только указывают. Это и есть смысл структуры — хранить каждую сущность ровно один раз.

  *(практика: найти в Конфигураторе справочник «Контрагенты» — сколько реквизитов? Что было бы, если бы каждый документ хранил все эти данные самостоятельно?)*

- **§ 1.6.** Чтение справочника: перевести на бизнес-язык *(практика: открыть справочник, прочитать вслух как описание бизнес-сущности)*
- **§ 1.7.** Создаём справочник «Товары» для «Канцтоваров» *(практика: создать объект в Конфигураторе)*

*Щёлкает:* читатель понимает, зачем нужен справочник как отдельный объект — и видит первый инсайт: структура существует, чтобы каждая сущность жила ровно в одном месте.

### Модуль 2. Глаголы — документы

- **§ 2.1.** Документ как запись факта (поступление, продажа, оплата)
- **§ 2.2.** Реквизиты документа: номер, дата, организация, контрагент
- **§ 2.3.** Табличная часть документа — список позиций
- **§ 2.4.** Проведение документа *(мимоходом: «документ может писать в регистр — вернёмся в Модулях 3 и 4»)*
- **§ 2.5.** Зачем ссылка: проблема хранения строки

  Реквизит «Контрагент» в документе можно хранить как строку — просто написать «ООО Ромашка». Но тогда при переименовании контрагента все старые документы будут показывать старое название. Исправить вручную — тысячи строк. Нужно другое решение: хранить не копию имени, а **указатель** на запись в справочнике. Тогда имя берётся из справочника каждый раз заново. *(практика: открыть документ в демо-базе, найти реквизит-контрагент, нажать на него — он открывает справочник)*

- **§ 2.6.** Как работает ссылка: имя собственное в палитре типов

  Тип реквизита «Контрагент» — не Строка, а «СправочникСсылка.Контрагенты». Это **имя собственное**: не описание, а адрес конкретного объекта в дереве. За ним стоит живой объект со всеми реквизитами. Сравнение: примитивный тип хранит значение само по себе — ссылочный тип хранит адрес объекта в системе.

  > Ссылочный тип — это имя собственное. Примитивный тип — описание. Имя указывает на объект; описание содержит смысл само по себе.

  *(практика: открыть палитру типов у реквизита в Конфигураторе — увидеть там все созданные справочники как доступные типы)*

- **§ 2.7.** Объект = тип: создав справочник, ты создал не только список

  К этому моменту читатель видел два объекта: справочник «Товары» и документ «Поступление». Документ ссылается на справочник — использует его как тип реквизита. Значит, когда ты создаёшь справочник, ты создаёшь не только список: ты добавляешь в конфигурацию **новый сложный тип данных**, которым могут пользоваться другие объекты. Это принципиальное отличие от примитивных типов — они существовали до тебя. Ссылочный тип ты создаёшь сам, добавляя объект в дерево.

  > Дерево метаданных — это одновременно структура данных и система типов конфигурации. Каждый новый объект в дереве — новый тип.

  *(практика: открыть палитру типов — убедиться, что созданный справочник «Товары» там есть)*

- **§ 2.8.** Создаём документ «ПоступлениеТоваров» для «Канцтоваров» *(практика)*

*Щёлкает:* читатель понимает зачем нужна ссылка, видит её в Конфигураторе — и осознаёт, что каждый объект дерева становится новым типом данных.

### Модуль 3. Таблицы фактов — регистры сведений

Регистр сведений — самый простой вид регистра: таблица, которая хранит факты. Никаких движений, никаких итогов. Только: «на такую-то дату, для такого-то объекта — такое-то значение». Именно с него начинается понимание всех остальных регистров.

- **§ 3.1.** Зачем нужны регистры: хранить то, что нельзя вычислить из документов
- **§ 3.2.** Регистр сведений независимый: курсы валют — данные вводятся вручную, без документа *(практика — посмотри и ответь: найти независимый регистр сведений в демо-базе; почему здесь нет регистратора? Что было бы, если бы курс валюты хранился как реквизит справочника «Валюты»?)*
- **§ 3.3.** Измерения и ресурсы: таблица как «ключ → значение». Сначала — простая таблица из двух колонок. Потом: ключ = измерения (по чему храним), значение = ресурс (что именно). *(практика — посмотри и ответь: открыть регистр сведений, найти измерения и ресурсы — что будет, если добавить второе измерение?)*
- **§ 3.4.** Периодичность: как часто меняется значение *(практика — посмотри и ответь: найти в демо-базе регистр с разной периодичностью — что изменится в поведении регистра, если поставить «непериодический»?)*
- **§ 3.5.** Регистр сведений подчинённый регистратору: Приказ об изменении цен пишет в регистр «ЦеныТоваров» *(практика — посмотри и ответь: найти вкладку «Движения» у документа-регистратора — что произойдёт с историей цен, если удалить документ?)*
- **§ 3.6.** Понятие регистратора: документ берёт на себя ответственность за запись в регистр
- **§ 3.7.** Создаём регистр «ЦеныТоваров» и документ «ПриказОбИзмененииЦен» для «Канцтоваров» *(практика)*

```
Регистр сведений «ЦеныТоваров»
├── Измерения: Товар (ключ)
├── Ресурсы: Цена
├── Периодичность: По позиции регистратора
└── Регистратор: ПриказОбИзмененииЦен
```

*Щёлкает:* читатель понимает регистр сведений как прилагательное во времени и видит роль регистратора — документа, который фиксирует не только факт изменения, но и его причину.

### Модуль 4. Амбарные книги — регистры накопления

Регистр накопления — это регистр сведений, которому добавили направление. Каждая запись теперь имеет знак: приход или расход. Из этого появляются итоги — остатки и обороты. Понятие регистратора остаётся тем же: документ пишет движение.

- **§ 4.1.** От сведений к накоплению: что меняется, когда добавляем направление
- **§ 4.2.** Движение: приход и расход — две стороны одной записи
- **§ 4.3.** Измерения и ресурсы в регистре накопления (товар, склад → количество)
- **§ 4.4.** Остатки и обороты: два вида итогов, два вида вопросов
- **§ 4.5.** Регистратор накопления: документ «ПоступлениеТоваров» пишет движение «Приход» *(практика — посмотри и ответь: настроить вкладку «Движения» в Конфигураторе; что изменится в остатках, если убрать документ из регистраторов?)*
- **§ 4.6.** Чтение регистра накопления: «по каждому товару и складу — остаток» *(практика)*
- **§ 4.7.** Создаём регистр «ОстаткиТоваров» для «Канцтоваров» *(практика)*

```
Регистр накопления «ОстаткиТоваров»
├── Измерения: Товар, Склад
├── Ресурсы: Количество
├── Вид: Остатки
└── Регистраторы: ПоступлениеТоваров (Приход), РеализацияТоваров (Расход)
```

### Модуль 7. Другие объекты дерева

К этому моменту читатель знает все основные объекты. Модуль 7 — не «ещё куча сложного». Это **знакомые идеи в новых формах**: каждый объект здесь вырастает из уже понятого. Если что-то кажется сложным — значит, в нём есть знакомая основа плюс одно новое правило.

- **§ 7.1.** Перечисления: фиксированный набор значений (виды оплаты, статусы) *(практика: найти перечисление в демо-базе, ответить — почему здесь не справочник?)*
- **§ 7.2.** Регистры бухгалтерии: накопление плюс двойная запись

  Регистр бухгалтерии — это регистр накопления плюс одно новое правило: каждое движение одновременно пишется по дебету одного счёта и кредиту другого. Всё остальное — уже знакомо: измерения, ресурсы, регистратор. *(практика: найти ветку «Регистры бухгалтерии» в демо-базе — найти знакомые элементы: измерения, ресурсы)*

- **§ 7.3.** Регистры расчёта: сведения плюс периоды действия

  Регистр расчёта — это регистр сведений, подчинённый регистратору, плюс одно новое: периоды действия. Каждая запись расчёта действует за конкретный период и может зависеть от другой записи. Типичный пример — начисление зарплаты. *(практика: найти ветку «Регистры расчёта» в демо-базе — найти знакомые элементы)*

- **§ 7.4.** Планы видов характеристик: прилагательные, которые придумывает пользователь

  К этому моменту читатель знает три вида прилагательных: реквизит (задан разработчиком, статичен), регистр сведений (задан разработчиком, меняется во времени). Третий вид — **план видов характеристик**: прилагательные, которые разработчик не знал заранее и которые пользователь добавляет сам уже после запуска системы.

  Для ручки важен цвет, для бумаги — плотность, для папки — формат. Разработчик не может предусмотреть всё. Решение: создать «рамку» — план видов характеристик, — в которую пользователь вписывает нужные свойства. Значения хранятся в регистре сведений.

  ```
  Реквизит справочника     →  прилагательное разработчика (известно заранее)
  Регистр сведений         →  прилагательное во времени (известно заранее, меняется)
  План видов характеристик →  прилагательное пользователя (неизвестно на этапе разработки)
  ```

  *(практика: найти ветку «Планы видов характеристик» в демо-базе — ответить: где хранятся значения характеристик? Найти связанный регистр сведений)*

- **§ 7.5.** Планы счетов и планы обмена *(мимоходом)*
- **§ 7.6.** Бизнес-процессы и задачи
- **§ 7.7.** Роли: кто что может видеть — упоминание

*Щёлкает:* читатель видит незнакомые объекты — и находит в каждом знакомую основу плюс одно новое правило.

Этот модуль — самый практически ценный. Читатель открывает реальную типовую конфигурацию (например, демо-базу УТ или БП) и видит 500+ объектов. Задача — не испугаться и найти нужное.

## VI. Терминологические соглашения

| Термин | Значение |
|---|---|
| Дерево метаданных | Окно «Конфигурация» в Конфигураторе: корень — «Конфигурация», ветки — «Справочники», «Документы» и т.д. |
| Объект метаданных | Конкретный элемент в дереве: справочник «Товары», документ «Поступление» |
| Реквизит | Свойство объекта — данные, которые хранятся |
| Табличная часть | Список строк у объекта; каждая строка — набор реквизитов |
| Примитивный тип | Тип данных, хранящий значение само по себе: Строка, Число, Дата, Булево |
| Ссылочный тип | Тип данных, указывающий на объект метаданных: СправочникСсылка, ДокументСсылка и др. |
| Измерение | В регистре: аналитический разрез (по чему храним) |
| Ресурс | В регистре: числовое значение (количество, сумма) |

### Порядок ввода терминов

- Модуль 0 — только «дерево», «объект», «конфигурация», «Конфигуратор»; одна мысль про модули («хранится / делает»).
- Модуль 1 — «справочник», «реквизит», «табличная часть», «примитивный тип» (§1.2, только как свойство реквизита — без философии системы типов).
- Модуль 2 — «документ», «проведение» (мимоходом); «зачем ссылка» (§2.5); «ссылочный тип / имя собственное» (§2.6); «объект = тип» (§2.7, когда читатель уже видел оба объекта и их связь).
- Модуль 3 — «регистр сведений», «измерение», «ресурс», «периодичность», «регистратор» (полностью).
- Модуль 4 — «регистр накопления», «движение», «остатки», «обороты».
- Модуль 7 — «план видов характеристик» (§7.4, полноценный параграф с цепочкой реквизит → регистр → ПВХ).
- Модули 5–9 — остальное.

## VII. Якорные правила

Правила разделены на три уровня. Базовые повторяются на протяжении всего курса. Второй уровень вводится по мере прохождения. Третий — расширения для тех, кто идёт дальше.

### Базовые (повторяются постоянно)

> Дерево метаданных — это чертёж системы, а не программа.

> Справочник — «кто/что», документ — «что произошло», регистр сведений — «каково значение во времени», регистр накопления — «сколько в итоге».

> Один бизнес-процесс — одна цепочка: справочник → документ → регистр → отчёт.

> Нельзя понять дерево метаданных, не понимая бизнес-задачи. И наоборот.

### Второй уровень (вводятся по ходу курса)

> Есть то, что хранится — дерево метаданных. Есть то, что делает — модули. Мы работаем с первым.

> Примитивный тип хранит значение само по себе. Ссылочный тип — имя собственное: за ним стоит живой объект со всеми свойствами.

> Создав объект метаданных, ты создал новый сложный тип данных. Каждая ветка дерева — новый тип.

> Регистратор — документ, который берёт ответственность за запись в регистр. Он фиксирует не только факт изменения, но и его причину.

> Читать большое дерево нужно не сверху вниз, а от задачи к объекту.

### Третий уровень (расширения)

> Регистр накопления — это регистр сведений с направлением. Регистр бухгалтерии — накопление с двойной записью. Регистр расчёта — сведения с периодами действия.

> Реквизит = прилагательное разработчика. Регистр сведений = прилагательное во времени. План видов характеристик = прилагательное пользователя.

> Если вы видите незнакомый тип объекта — начните с вопроса «зачем он мог понадобиться автору».
